================================================================================
KEY FEATURES IMPLEMENTED
================================================================================

SPAWN FEE SYSTEM:
  • Minimum fee: 1 token (configurable)
  • Maximum fee: 1000 tokens (configurable)
  • Validates wallet address format
  • Creates payment record before charging
  • Tracks transaction hash and status
  • Supports both blocking and async patterns

REVENUE DISTRIBUTION:
  • Host receives 50% of spawn fee
  • Participants receive 40% (split equally)
  • Platform receives 10%
  • Example: 1000 tokens → 500 host + 400 participants + 100 platform
  • Supports 0, 1, or many participants
  • Precision maintained with BigInt (no float rounding)

PAYMENT STATUS TRACKING:
  • Pending → pending payment confirmation
  • Confirming → awaiting block confirmation
  • Confirmed → transaction confirmed onchain
  • Failed → insufficient balance, rate limit, network error
  • Automatically updates from x402 API and webhooks

WEBHOOK SECURITY:
  • HMAC-SHA256 signature verification
  • Signature from x-x402-signature header
  • Prevents webhook tampering
  • Prevents replay attacks
  • Returns 401 for invalid signatures

ERROR RECOVERY:
  • INSUFFICIENT_BALANCE → Mark failed, return 402
  • RATE_LIMIT → Retry with exponential backoff
  • NETWORK_ERROR → Queue for retry
  • Graceful degradation if x402 unavailable

ARCHITECTURE INTEGRATION
================================================================================

PAYMENT FLOW:
  1. User initiates room creation
     ↓
  2. chargeSpawnFee(agentId, wallet, roomId)
     • Validates wallet address
     • Creates payment record (status: pending)
     • Calls x402Client.createPayment()
     • Returns payment with transaction hash
     ↓
  3. Payment status polling
     • checkPaymentStatus(paymentId)
     • Query x402 API every 5 seconds
     • Update database when status changes
     OR
     • Receive webhook from x402
     • Verify HMAC signature
     • Update status immediately
     ↓
  4. Payment confirmed
     • Create room
     • Start orchestration
     • Begin audio streaming
     ↓
  5. Room complete
     • Trigger distributeRevenue()
     • Calculate splits (50/40/10)
     • Create payout transactions
     ↓
  6. Revenue distributed
     • Host receives 50%
     • Participants receive 40% (split)
     • Platform receives 10%
