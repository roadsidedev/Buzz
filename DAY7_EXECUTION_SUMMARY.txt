================================================================================
KEY DECISIONS & RATIONALE
================================================================================

1. Singleton Pattern for Services
   - Rationale: Single instance per runtime, consistent state
   - Benefits: Memory efficient, predictable lifecycle

2. Separate Message Service from Turn Management
   - Rationale: Single responsibility principle
   - Benefits: Message operations can be used independently
   - Flexibility: Easy to test message operations separately

3. Output Contract as Separate Service
   - Rationale: Domain-specific completion logic
   - Benefits: Contracts centralized, easy to update
   - Flexibility: Contracts per room type cleanly separated

4. Turn Loop Interval-Based (not Event-Based)
   - Rationale: Simpler initial implementation
   - Benefits: Predictable behavior, debugging easier
   - Trade-off: May check for messages when none available

5. Fallback Scoring (FALLBACK_SCORE = 50)
   - Rationale: Graceful degradation if orchestrator down
   - Benefits: Room continues even if orchestrator fails
   - Safety: Score 50 means message is neutral (not auto-selected)

6. Message Status State Machine
   - States: candidate → queued → selected → playing → played
   - Benefits: Clear lifecycle, prevents invalid transitions
   - Flexibility: Can handle rejection at any state
